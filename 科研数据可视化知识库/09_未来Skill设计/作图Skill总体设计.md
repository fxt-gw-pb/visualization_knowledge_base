# 作图 Skill 总体设计

> 说明本知识库**如何长成**一个 R/Python 双后端的自动作图 Skill。当前阶段先把知识库做对做全；本文是“未来形态”的蓝图，倒推现在该怎么沉淀知识。

## 1. 愿景一句话

> 用户给「数据 + 研究问题 + 用途」，Skill 自动**选图 → 选后端 → 选配色/架构 → 生成可运行 R/Python 代码 → 出图 → 自检返修 → 导出**，质量逐步逼近顶刊。

## 3. Skill 的最小能力边界（先做什么、不做什么）

✅ 做：
- 根据数据类型 + 问题，从 12 个 MVP 图里选型并出图。
- 双后端生成可运行代码（不直接黑盒出图，让用户可复现/可改）。
- 套用本库配色/架构/导出规范。
- 出图后跑质检清单，自动返修常见问题。

❌ 暂不做（MVP 之外）：
- 任意自定义复杂图（先覆盖高频科研图）。
- 自动统计建模决策（建模由用户/统计模块定，Skill 负责“把结果画好”）。
- 交互式图（先静态发表图）。

## 4. 关键设计原则

1. **知识与代码分离**：图表知识在 Obsidian 卡片，Skill 读卡片而非硬编码，便于迭代。
2. **代码透明可复现**：Skill 产出**脚本**而非只产图，用户能看懂、能改、能复现（[[R导出规范]]/[[Python导出规范]]）。
3. **配色不写死**：引用 registry 名（`cat_main`/`seq_viridis`…），改色改一处。
4. **质检内建**：每张图必过 [[科研图表质量检查清单]] + [[图表返修规则总表]]。
5. **双后端对等**：不偏袒，按 [[R与Python后端选择规则]] 路由。
6. **状态驱动**：只用 `reviewed`/`tested` 以上的卡片做生产，`draft` 仅供探索。

## 5. 从知识库到 Skill 的演进路线（概览，详见 [[MVP路线图]]）

```text
阶段 0（现在）：知识库 + 12 MVP 卡片 + Framingham 测试区
阶段 1：把卡片 frontmatter + 模板抽成机器可读 spec（[[输入输出规范]]）
阶段 2：写决策器（[[图表决策流程]]）+ 后端路由 + 代码模板渲染
阶段 3：接质检 + 返修后处理（[[科研图表质量检查清单]]/[[图表返修规则总表]]）
阶段 4：用 Framingham 任务清单做回归测试，逐图升 tested
阶段 5：扩展到全部图表族，接更多数据集
```

## 6. 给“现在”的反推要求

为了未来 Skill 顺利，**现在写卡片就要做到**：

- frontmatter 字段齐全且规范（机器可读）。
- 代码模板标 `# >>> PARAM` 注入点。
- 配色引用 registry 名。
- 返修规则写成 IF→THEN。
- 每图绑定 Framingham 测试任务。
